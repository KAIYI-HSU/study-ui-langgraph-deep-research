1. 專案總覽與定位

此 repo 是 Gemini Fullstack LangGraph Quickstart 的「增強分支」(Enhanced Version),作者為 Peng.G。 原始版本展示了一個「以 LLM 反思 (Reflection) 為核心」的單階段研究 Agent;本版本則將其升級為:

  • 多任務規劃 (Multi-Task Planning):由 planner_node 先把使用者問題拆解成多個子研究任務,逐一執行。
  • 雙層評估 (Reflection + Enhanced Evaluation):在反思之外再加一層 evaluate_research_enhanced 把「是否繼續研究」「是否需要 Firecrawl 增強」獨立成決策邊。
  • 內容深度增強 (Content Enhancement):整合 Firecrawl,對 Gemini Search 拿不到完整正文的關鍵 URL 做深度抓取。
  • 整體式報告合成 (Holistic Integrated Report):由 task-by-task 拼接改為單次大型 LLM 合成,提升敘事一致性。
  • 前端 Think Panel:在右側獨立面板呈現 Plan、各 Task、各 Step (查詢生成 / 網路研究 / 反思 / 增強 / 評估 / 完成) 的層次化視覺化,並以 sessionStorage 累積事件,離線可重算。
核心觀察:此專案的價值不僅在於「能跑」,而在於它示範了一個常見的 LangGraph 痛點解法 —— 如何把純 graph state 的事件流,轉成使用者可理解的「計畫 → 任務 → 步驟」階層。這個轉換邏輯集中在 frontend/src/utils/dataTransformer.ts,是本 repo 最具參考價值的部分。

3.2 狀態模型 OverallState

用 LangGraph 的 Annotated[list, operator.add] 累加器來支援平行分支的結果合併,是這類 fan-out / fan-in 圖的標準做法。

class OverallState(TypedDict):
    messages: Annotated[list, add_messages]      # 對話歷史
    user_query: str
    plan: list                                   # planner 輸出的任務列表
    current_task_pointer: int                    # 當前任務索引
    executed_search_queries: Annotated[list, operator.add]
    web_research_result: Annotated[list, operator.add]
    sources_gathered: Annotated[list, operator.add]
    research_loop_count: int

    # Day-2:多任務迭代
    ledger: Annotated[List[LedgerEntry], operator.add]
    global_summary_memory: Annotated[List[str], operator.add]

    # Day-3:富合成
    current_task_detailed_findings: Annotated[List[Dict], operator.add]
    task_specific_results: Annotated[List[Dict], operator.add]
    final_report_markdown: Optional[str]

    # Reflection 結果
    reflection_is_sufficient: Optional[bool]
    reflection_knowledge_gap: Optional[str]
    reflection_follow_up_queries: Optional[List[str]]

    # 強化評估
    evaluation_is_sufficient: Optional[bool]
    evaluation_should_continue: Optional[bool]
    evaluation_follow_up_queries: Optional[List[str]]
    evaluation_research_complete: Optional[bool]

    # 內容增強
    enhancement_decision: Optional[Dict]
    enhancement_status: Optional[str]   # skipped/completed/failed/error
    enhanced_content_results: Optional[List[Dict]]
    enhanced_sources_count: Optional[int]

3.3 Graph 拓樸

START └─▶ planner ──▶ generate_query ──┐ (conditional fan-out) ├─▶ web_research (Send × N) │ │ │ ▼ │ reflection │ │ │ ▼ should_enhance_content? │ ┌───┴───────────────┐ │ ▼ ▼ │ content_enhancement │ │ │ │ │ └────▶ evaluate_research_enhanced │ │ decide_next_research_step │ ┌──────────────┴──────────┐ │ ▼ (follow-up Send) ▼ └ web_research record_task_completion │ ▼ decide_next_step_in_plan ┌──────────┴──────────┐ ▼ ▼ generate_query finalize_answer (下一個任務) │ ▼ END

3.4 關鍵節點解析

  • planner_node:用結構化輸出 (ResearchPlan Pydantic) 一次取得多個 ResearchTask,設定 current_task_pointer=0。
  • generate_query:依當前 task 的 description 產生 N 個查詢(Effort 控制:low=1, medium=3, high=5)。
  • continue_to_web_research:條件邊用 Send 物件平行 fan-out N 個 web_research 子節點。
  • web_research:用 google.genai.Client 原生(非 LangChain)呼叫以取得 grounding_metadata,做 URL resolve + 引用插入。
  • reflection:嘗試 structured output(Reflection),失敗時 fallback 為 text + regex JSON,再 fallback 為「>= 3 筆視為充足」的硬規則。
  • content_enhancement_analysis:LLM 判斷是否需要 Firecrawl 抓內文;若 FIRECRAWL_API_KEY 缺則直接 enhancement_status = "skipped_no_api"。
  • evaluate_research_enhanced:把 reflection 結論與「增強加分」(每增強源 +0.3,封頂 0.8)合成最終 evaluation_is_sufficient。
  • record_task_completion_node:把當前 task 的發現摘要寫入 ledger,清空 current_task_detailed_findings,指針 +1。
  • finalize_answer:統合所有 ledger + task_specific_results,先呼叫 integrate_report_enhancement_into_finalize 做最後一層增強,再用 tiktoken 切到 120k tokens 內,單次 LLM 整合輸出。最後做清理(clean_generated_content / remove_prompt_remnants / final_quality_check)。

3.5 容錯設計亮點

  • Reflection 有三層 fallback,確保不會卡住整個圖。
  • Web research 失敗時仍會回傳 error_content 並帶上 task_id,讓下游不丟資料。
  • 引用清理:clean_malformed_citations + final_quality_check 處理 Vertex AI 短網址污染。
  • Token 預算:split_by_tokens 用 cl100k_base 計數,提取 factual / critical section 優先保留。

5. LangGraph 串接 UI:Plan / Task 呈現機制

5.1 串接的「五個關鍵接觸點」

(A) useStream hook 建立串流

const thread = useStream<{
  messages: Message[];
  initial_search_query_count: number;
  max_research_loops: number;
  reasoning_model: string;
}>({
  apiUrl: import.meta.env.DEV ? "http://localhost:2024" : "http://localhost:8123",
  assistantId: "agent",         // 對應 langgraph.json 的 graphs.agent
  messagesKey: "messages",       // 告訴 SDK 要把哪個 state 鍵當 messages
  onFinish: (state) => { ... },
  onUpdateEvent: (event) => { ... }  // 每個節點完成都會呼叫
});

SDK 內部會呼叫 POST /threads/{id}/runs/stream,以 SSE 把每個節點的 state delta 推回,封裝為 { [nodeName]: { ...stateUpdate } } 格式。

(B) onUpdateEvent:節點 → 時間線項目的映射

if (event.generate_query) {
  processedEvent = { title: "Generating Search Queries",
                     data: event.generate_query.query_list?.join(", ") };
} else if (event.web_research) {
  processedEvent = { title: "Web Research",
                     data: `Gathered ${n} sources. Related to: ${labels}` };
} else if (event.reflection) {
  processedEvent = { title: "Reflection",
                     data: reflectionData.reflection_is_sufficient
                       ? "Search successful..."
                       : `Need more info: ${followUps.join(", ")}` };
} else if (event.planner_node || event.planner) { ... }
else if (event.content_enhancement) { ... }
else if (event.evaluate_research_enhanced) { ... }
else if (event.record_task_completion) { ... }
else if (event.finalize_answer) { hasFinalizeEventOccurredRef.current = true; }

(C) sessionStorage 累積完整事件

每個 onUpdateEvent 都把事件物件 push 進 sessionStorage 的 research_events 陣列。這支援:

  • 頁面重新整理仍能重建 ThinkPanel(同 tab 內)
  • 透過 useMemo 重算 transformEventsToHierarchy,不必額外維護組件狀態樹
  • 送 handleSubmit 時會清空 (sessionStorage.removeItem('research_events'))

(D) dataTransformer:扁平事件 → 階層結構

這是整個 UI 串接邏輯的核心。轉換器做三件事:

  1. 合併最新 state:遍歷所有事件,逐 key 把物件值淺合併到 latestState(因為 LangGraph 累加器讓每個事件只帶 delta)。
  2. 建立 Planning 區塊:從 state.plan 取 task 列表、用 current_task_pointer 標記當前任務。
  3. 建立 TaskDetail × StepCard:對「指針 ≤ 當前」的每個 task,從事件流中過濾出對應的 generate_query / web_research / reflection / content_enhancement / evaluate_research_enhanced / record_task_completion,組成 steps: TaskStep[]。

(E) ResearchThinkPanel 自動展開策略

  • 進行中:展開「有步驟」的任務,逐一顯示 Step。
  • 完成 (overallStatus === 'completed'):展開所有任務 + 所有步驟,讓使用者可回顧。
  • useEffect 依 currentTaskId / overallStatus / tasks.length 觸發,避免不必要的重渲。

5.2 重要的工程細節

議題本 repo 的做法備註
節點名 → UI 標題硬編碼 if/else 比對 event 的第一層 key缺點:後端改名要同步改前端;優點:零依賴、易讀
State 全量 vs Delta把事件 delta 逐次 shallow-merge,假設累加器(operator.add)只增不減對於非累加欄位(如 current_task_pointer)新值直接覆寫舊值
歷史快照每當看到 Reflection / Enhancement / Evaluation 事件,延遲 300ms 後把當前 timeline 存到 historicalActivities[messageId]避免 SDK 還沒把 AI 訊息建好就拿空 ID
Effort → Loop/Query 換算前端 submit 前把 low/medium/high 映射成 (1,1)/(3,3)/(5,10)等於把 UI 概念翻譯成後端 Configuration 期待的數值
取消執行thread.stop() + window.location.reload()實作上是「硬重整」,代價是失去當前 messages,但簡單可靠

6. Human-in-the-Loop 機制分析

誠實的觀察:此 repo 並未使用 LangGraph 原生的 interrupt() / Command(resume=...) 真正 HITL API。它的「人類介入」是 pre-run 配置 + run-time 取消,屬於弱形式 HITL。

6.1 已實作的人機互動

  • Pre-run 配置:Effort 等級、Reasoning Model 選擇(InputForm.tsx)。
  • Mid-run 中止:thread.stop() 透過 SDK 呼叫 LangGraph Server 的取消端點。
  • 視覺化監督:Think Panel 即時呈現決策路徑,使用者可知 Agent「為什麼這樣做」。

6.2 缺少的真 HITL

  • 無 interrupt_before / interrupt_after 設定 —— 不會在某節點前暫停等使用者確認。
  • 無 Command(resume=human_input) —— 無法在執行中採納使用者修正並繼續。
  • Plan 是 LLM 全自動產生,使用者不能改 task 列表後再執行。

6.3 如要加入真 HITL,最小可行改動

# backend: graph.compile 時加入
graph = builder.compile(
    name="pro-search-agent",
    interrupt_after=["planner"],   # 計畫產生後暫停
)

# 前端:useStream 提供 .interrupt 與 .submit({command:{resume:...}})
if (thread.interrupt) {
  // 顯示 plan 編輯 UI,使用者按 "Approve / Edit"
  thread.submit(undefined, { command: { resume: editedPlan } });
}

7. 從 LangGraph 串接 UI 的完整生命週期

以下為使用者在輸入框送出問題 → 看到最終 markdown 報告的完整 12 個步驟,涵蓋網路、狀態、視覺三條時間線。

Step 1. 使用者操作 — 在 InputForm 輸入問題,選 Effort=medium、Model=gemini-2.5-flash。按 Send 觸發 handleSubmit("...","medium","gemini-2.5-flash...")。
Step 2. 前端轉換 → SDK submit — App.tsx 把 effort 映射成 (initial_search_query_count=3, max_research_loops=3),清空 sessionStorage 的 research_events,呼叫 thread.submit({ messages, initial_search_query_count, max_research_loops, reasoning_model })。
Step 3. SDK HTTP 呼叫 — @langchain/langgraph-sdk 內部建立 thread(若無),向 POST /threads/{id}/runs/stream 送出。請求中 assistantId="agent" 對應 langgraph.json 的 graphs.agent。
Step 4. 後端 StateGraph 啟動 — LangGraph Server 注入 OverallState 初始值,從 START 進入 planner_node。Gemini 用 ResearchPlan structured output 回傳 N 個 ResearchTask,設定 plan 與 current_task_pointer=0。SSE 推送第一個事件 { planner: { plan: [...], user_query, current_task_pointer: 0 } }。
Step 5. 前端首次 onUpdateEvent — 命中 event.planner_node || event.planner 分支,push {title:"Planning Research Strategy", data:"Generated N research tasks"} 進 processedEventsTimeline;同時把整個 event 物件 push 進 sessionStorage。
Step 6. dataTransformer 首次重算 — ChatMessagesView 的 useMemo 依賴變更,重新呼叫 transformEventsToHierarchy。發現 state.plan 已存在,建立 planning 區塊與 tasks: TaskDetail[](目前只有 task[0] 標為 in_progress)。ResearchThinkPanel useEffect 觸發,展開 task[0]。
Step 7. 進入 generate_query → fan-out — 後端對 task[0].description 呼叫 SearchQueryList structured output,推 { generate_query: { query_list: [...] } }。continue_to_web_research 條件邊用 Send 平行 fan-out 3 個 web_research 子節點(每個帶 search_query + current_task_id)。
Step 8. 平行 web_research 事件流 — 後端對每個查詢呼叫 genai_client.models.generate_content(tools=[google_search]),取 grounding metadata 與引用插入。前端會收到 3 個獨立的 { web_research: { sources_gathered: [...], executed_search_queries: [...] } }。dataTransformer 把每個事件展開為一個 web_research Step,顯示 query + sources 卡片。
Step 9. reflection → 條件路由 → enhancement (可選) — 後端呼叫 reflection_instructions,得 {reflection_is_sufficient, knowledge_gap, follow_up_queries}。前端會看到一個 { reflection: {...} } 事件。接著 should_enhance_content 邊判斷是否進 Firecrawl;若進,前端再收一個 { content_enhancement: { enhancement_status, enhanced_sources_count } }。
Step 10. evaluate_research_enhanced → 路由決策 — 後端合成 evaluation_is_sufficient,推事件給前端;同時 decide_next_research_step 條件邊決定回 web_research(follow-up Send)或前進到 record_task_completion。Snapshot 機制此時觸發,把當前 timeline 存到 historicalActivities。
Step 11. record_task_completion → 下一個 task 或 finalize — 後端把 task[0] 的 ledger entry 寫入,指針 +1,清空 current_task_detailed_findings。decide_next_step_in_plan 若 pointer < len(plan) 回到 generate_query(進入 task[1]),否則進 finalize_answer。前端 dataTransformer 在下個 generate_query 事件來時,自動把 task[0] 標 completed、task[1] 標 in_progress。
Step 12. finalize_answer + onFinish — 所有 task 跑完後,finalize 整合 ledger / task_specific_results / report-level enhancement,單次 LLM 產出 markdown,清理引用後寫入 messages(AIMessage)。前端收到 { finalize_answer: {...} } 設 hasFinalizeEventOccurredRef = true;thread.messages 更新觸發 useEffect 把當前 timeline 永久存到 historicalActivities[lastMessage.id]。ChatMessagesView 渲染 ReactMarkdown,Activity Timeline 自動摺疊。

7.1 事件流時序圖

時間 ──▶ User submit ─┐ ▼ thread.submit (SDK) │ POST /runs/stream │ ├─[event] planner ─▶ ThinkPanel:planning 區塊建立 ├─[event] generate_query ─▶ Step: "Generating Search Queries" ├─[event] web_research × 3 ─▶ 3 × Step: "Web Research: ..." ├─[event] reflection ─▶ Step: "Reflection Analysis" ├─[event] content_enhancement ─▶ Step: "Content Enhancement Analysis" ├─[event] evaluate_research... ─▶ Step: "Research Quality Evaluation" ├─[event] record_task_completion─▶ Step: "Task Completion Recorded" │ ↻ (回到 generate_query 處理下一個 task) ├─[event] generate_query (task[1]) ├─... ├─[event] finalize_answer ─▶ ActivityTimeline 摺疊 └─[message] AIMessage(content=markdown) ─▶ ReactMarkdown 渲染

8. 優缺點評估

✔ 優點

  • 多任務規劃:把單問題拆成可獨立追蹤的子任務,顯著提升複雜研究問題的答案結構。
  • 容錯設計成熟:reflection 三層 fallback、web_research try/except、Token 預算控制,在 LLM 不穩定時仍可完成 run。
  • UI 思考面板:把 LangGraph 內部事件視覺化為「計畫 → 任務 → 步驟」階層,可解釋性遠高於原始 chat UI。
  • SDK 整合乾淨:用 useStream 一個 hook 解決串流 / 訊息 / 取消,前端不必自己處理 SSE。
  • Firecrawl 增強:對 Google Search 拿不到正文的場景,提供 graceful 升級路徑;沒 API Key 也能無痛跳過。
  • 引用系統:把 Vertex AI 短網址換成可驗證的 [Source: Domain (URL)],提升可信度。

✘ 缺點與技術債

  • 真 HITL 缺失:沒有 interrupt 機制,使用者不能在 plan 階段修正任務。
  • 前後端耦合在事件 key:後端節點名一改,前端的 if/else 大鏈條與 dataTransformer 都要同步改。沒契約層。
  • graph.py 過大 (1221 行):節點、邊、引用清理工具、token 工具全擠一檔,違反單一職責。
  • 大量 debug console.log:App.tsx / dataTransformer.tsx / ResearchThinkPanel.tsx 留下生產不該有的 log,且部分仍用簡體中文。
  • 本地化混亂:後端 print 與部分 reflection prompt 混用中文,UI 文字英文,文檔中英 + 簡繁混用。
  • useStream 的 apiUrl 硬編碼:Dev/Prod 用 import.meta.env.DEV 切換,沒走環境變數,自訂部署不便。
  • 取消 = 整頁 reload:window.location.reload() 太粗暴,會丟失對話歷史。
  • SessionStorage 容量風險:長 run 累積大量事件可能撞上 5MB 上限,且 tab 關閉即失。
  • finalize 單次 LLM 120k tokens:成本高,且超量時直接截斷可能丟資訊。
  • Reflection 默認「3 筆即充足」:Fallback 過於樂觀,可能未達深度即停止研究。

10. 結論與建議

此 repo 是研究「LangGraph 多階段 Agent ↔ React 即時視覺化」的優良參考。其核心價值有三:

  1. 把抽象 graph 狀態轉成使用者可懂的階層:這是 LangGraph 應用最普遍的工程難題,本 repo 的 dataTransformer + ResearchThinkPanel 模式可直接複用。
  2. 多層次的容錯:LLM structured output 不穩、Google Search 偶爾無結果、Token 爆量,均有對應 fallback,是生產級 Agent 的縮影。
  3. 「弱 HITL」的 UX 設計:用 Effort、Model、可解釋面板把使用者納入觀察迴圈,即使沒實作 interrupt() 也提供了相當程度的人類監督。

若要把本專案推向生產,建議優先:

  • 引入 LangGraph 原生 interrupt_after=["planner"] 讓使用者編輯 plan。
  • 把 onUpdateEvent 的 if/else 重構為「節點名 → 處理函式」的字典,並抽到 eventRegistry.ts。
  • 清理 console.log 與中文 print;統一語系。
  • graph.py 拆成 nodes/、edges/、utils/ 三個子模組。
  • 用 IndexedDB 取代 sessionStorage 以支援長 run。
  • 取消改為呼叫 SDK thread.stop() 之後等待 onFinish,而非整頁 reload。
